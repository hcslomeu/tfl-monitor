# Napkin Runbook

Repo-specific runbook. Read first thing every session and curate. High-signal
recurring guidance only â€” not a session log.

## Curation Rules

- Re-prioritize on every read.
- Keep recurring, high-value notes only.
- Soft cap ~10 items per category: when a category exceeds it, prune or merge the lowest-value item on the next read. Favour recording a fresh high-value lesson now over hard-blocking; trim on the next curation pass.
- Each item includes date + "Do instead".

1. **[2026-04-26 / re-confirmed 2026-04-29 / re-confirmed 2026-05-19] Parallel sessions clobber the parent checkout â€” `isolation: "worktree"` is NOT enough; explicit `git worktree` is the only safe mitigation**
   Confirmed three times. Round 1 (TM-D5 PR #44): two `Agent` teammates spawned with `isolation: "worktree"` shared `/Users/humbertolomeu/tfl-monitor` and committed directly into the parent branch; agent-core's `f107802` landed before data-layer's three commits, saved only by disjoint file sets. Round 2 (PRs #87/#88, 2026-05-19): a sibling Claude session on `feature/perf-api-lazy-imports` repointed `feat/deploy-retry-budget`'s local ref, leaked stashes across branches, and reverted unstaged edits to `src/api/*.py` mid-task. Round 3 (PR #89, same day): web-track session saw its working tree mutated by the concurrent API-track session; `EnterWorktree` was disabled by the background-job runtime, so no automatic isolation was available. `git worktree list` always shows a single worktree.
   Do instead: when â‰Ą2 sessions or teammates must run concurrently, each one MUST execute as its first step `git fetch origin && git worktree add --detach "$HOME/.claude/jobs/<jobid>/wt" <remote-sha>`, edit + commit inside that path, `git push origin HEAD:<branch>` (the explicit `origin` is required â€” `git push HEAD:<branch>` parses `HEAD:<branch>` as a remote name and fails), then `cd "$HOME" && git worktree remove --force "$HOME/.claude/jobs/<jobid>/wt"` (the `cd` is required â€” `git worktree remove` refuses to delete a worktree that contains the current shell cwd). The detached-HEAD form keeps the worktree immune to branch-ref repointing from peer sessions; pinning to `<remote-sha>` (not the local branch) keeps it immune to ref drift. Do NOT rely on `isolation: "worktree"` and do NOT trust `EnterWorktree` in background-job sessions â€” pinned to the main checkout by the runtime.

7. **[2026-05-18] `ExitWorktree` with `action: "remove"` after a squash-merge requires `discard_changes: true`**

11. **[2026-05-19] `docker compose restart <svc>` does NOT re-read `env_file` â€” use `up -d --force-recreate --no-deps <svc>`**
    After URL-encoding `@` chars in `/opt/tfl-monitor/.env`'s `DATABASE_URL` to fix the Supabase pooler password parse, `docker compose restart api` reported success but `/api/v1/status/live` kept returning `OperationalError: failed to resolve host '<password-fragment>@<rest>'`. `compose restart` only sends `SIGTERM` (then `SIGKILL` after the stop timeout) to the existing container â€” it does NOT recreate the container, so the environment baked into the container at `create` time still wins. The fresh `env_file` value lands on disk but never reaches the container's `/proc/<pid>/environ`; only a `create` step can re-evaluate compose's `env_file:` directive.
    Do instead: after editing any `.env` referenced via `env_file:` in `docker-compose*.yml`, run `docker compose -f <file> up -d --force-recreate --no-deps <svc>` (or `compose rm -fsv <svc> && compose up -d <svc>`) â€” not `compose restart <svc>`. `up --force-recreate` tears the container down and re-creates it from the compose spec, which re-reads `env_file` from disk; `restart` only signals the running process. Same trap applies to `environment:` literals built from `${VAR_NAME}` shell interpolation â€” recreate, never restart. Sanity check: `docker inspect <container> --format '{{range .Config.Env}}{{println .}}{{end}}' | grep <KEY>` shows what the container ACTUALLY has after recreate.

12. **[2026-05-19] Supabase pooler passwords with reserved URL chars (`@`, `:`, `/`, `#`, `?`) must be URL-encoded in `DATABASE_URL`**
    Supabase password rotation can produce passwords containing reserved URL delimiter characters. The standard pooler connection string `postgresql://postgres.<ref>:<password>@<host>:6543/postgres` parses an un-encoded `@` inside `<password>` as the user/host separator, so the parser sees `postgres.<ref>:<fragment-before-@>` as the user and `<fragment-after-@>@<actual-host>` as the hostname. psycopg then surfaces `OperationalError: failed to resolve host '<password-fragment-after-@>@<rest>'` â€” and the un-encoded password value leaks into the error log. Same trap with `:` (port separator), `/` (path separator), `#` (fragment), `?` (query).
    Do instead: when composing any `DATABASE_URL` from a Supabase-issued password, URL-encode the password segment with `urllib.parse.quote(password, safe='')` before string-formatting it into the URL â€” `@` â†’ `%40`, `:` â†’ `%3A`, `/` â†’ `%2F`, `#` â†’ `%23`, `?` â†’ `%3F`. The pooler accepts both encoded and raw chars in the user/db segments, but the password segment MUST be percent-encoded. Production hot-fix on the Lightsail box used an in-place Python script (`urlsplit` â†’ `quote(parsed.password, safe='')` â†’ `urlunsplit`) with `.env.bak-<timestamp>` rollback, and printed only metadata (`old_len`, `new_len`, `host_ok`) â€” never the password value. If the credential is already deployed, rotate it on Supabase first if any logs (CI, GHA, Logfire) captured the raw form; do not assume "the box is private" is enough.

1. **[2026-05-19] Supabase pooler (pgbouncer transaction mode) refuses `%(p)s IS NULL` â€” needs `%(p)s::text IS NULL` to disambiguate parameter type**
   `/api/v1/disruptions/recent` returned HTTP 500 in production with `psycopg.errors.AmbiguousParameter: could not determine data type of parameter $1` on the line `%(mode)s IS NULL`, while the exact same query worked fine against the local-dev Docker Postgres (direct 5432 connection).
